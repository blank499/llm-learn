# 向大模型提问的技巧：从直觉到框架

## 写在前面

这是一份对话整理。我并不是在写一篇教程告诉你"标准答案"，而是在记录一次跟一只 AI 助手边界上的探索——关于"怎么提问才能让大模型给出更准确、更深入的回答"。

整个过程从一个直觉出发，逐步被拆散、重组、验证，最后沉淀出一套可以复用的框架。它不是一个 checklist，它是一张地图。

---

## 第一部分：直觉——专业术语 + 丰富描述

### 1.1 起点

最早的想法很简单：提问的时候多塞一些专业术语，同时把场景描述得更丰富，模型回答的质量就会更高。

在 GPU 显存这个话题上，与其问"显存是什么"，不如问"HBM3 带宽和 GDDR7 在 LLM 推理场景下的瓶颈差异"。前者得到的是一段教科书级别的宽泛解释，后者得到的是精准的工程分析。

这个直觉是对的，但它只是一个经验层面的观察，还没有被拆解成可迁移的方法。

### 1.2 这个直觉的底层机制

经过分析，这个技巧的本质可以拆成两个独立的机制。

第一个是**词汇锚定**（Lexical Anchoring）。用高精度的术语（而非日常用词），在模型的参数空间中锁定一个特定的知识子网。"显存"锁定一个模糊区域，"HBM3 带宽"锁定一个精确坐标。精度越高，模型在知识检索时的信噪比越高。

第二个是**约束饱和**（Constraint Saturation）。用尽量多的上下文约束条件（场景、前提、边界、排除条件），逐步压缩模型的输出可能性空间，直到只剩一个高概率、高精准的回答路径。每个约束都是一个 filter，筛掉不相关的知识分支，保留下来的就是你要的东西。

两点汇总：这个方法本质上是从**信号精度**和**约束密度**两个方向减少模型回答的不确定性——不是让模型"想起更多"，而是让模型"想偏的概率更低"。

### 1.3 一个需要校正的理解偏差

这个直觉在表述上有一个细微但重要的偏差。原文是这么说的：

> "通过专业术语精准激活了大模型的某块向量空间，当专业术语越细激活向量空间内通过相似度找到 token 符合用户预期的概率越大，然后通过丰富的描述在空间内类似集合做交集计算得出来的结果被剪枝后越可能符合用户预期。"

这句话有问题的地方在于"激活了某块向量空间"。更准确的描述是：**影响了 attention 的分布重心**。模型的所有参数始终是"激活"的——attention 不是在开关知识，而是在决定当前上下文中的 token 应该更多关注哪些位置的表示。这不是二进制开关，而是连续的权重分布。

另外，输出也不是一步到位的搜索。每个 token 的选取不仅依赖与当前上下文的相似度，还依赖前一个 token 是什么。所以完整的路径应该是：

```
输入（术语）     ->  attention 重心锁定到高特异性子空间
输入（描述约束） ->  在子空间内划多条交线（约束面）
                      |
             约束满足的子多面体
                      |
       自回归逐步采样：每步缩窄方差，直到收敛
                      |
                    输出
```

## 第二部分：完整的技巧体系

从词汇锚定和约束饱和这两个底层机制出发，可以推导出一套完整的提问技巧体系。以下是八个技巧，按使用场景分类。

### 2.1 基础技巧

**技巧一：术语锚定**

用精准的技术名词代替模糊的日常用词。这是所有技巧的基础，其他技巧的有效性都依赖于这个前提。

| 模糊版 | 精准版 |
|--------|--------|
| 显存够不够用 | A100 80GB 上 HBM3 带宽 vs SRAM 容量的 tradeoff |
| 训练很慢 | 反向传播时 activation memory 导致显存溢出 |
| 模型效果不好 | MMLU 上比 baseline 低 3 个百分点，集中在 reasoning 类题目 |

**技巧二：约束饱和**

给的语境越丰富，模型越不需要猜你想问什么。每个约束条件都在向量空间中划一道线，最终输出被限定在所有划线围起来的区域内部。

### 2.2 策略级技巧

**技巧三：角色/视角指定**

指定模型应该从什么身份或视角回答问题。不同视角下模型调用的知识子网不同。硬件架构视角想到的是 HBM/GDDR/带宽延迟，编程视角想到的是 shared memory / coalesced access / bank conflict。

> 从 CUDA kernel 优化的角度，解释 GPU 显存层级和程序员需要注意什么。

**技巧四：声明已知基线（Known-Knowns）**

告诉模型你已经知道什么，让输出跳过基础层，直接进入你关心的深度。

> 我知道显存分 global/shared/local/register 四层，但想知道 SM 内部是怎么调度 shared memory 分配的。

**技巧五：要求推导链展开**

不只要结论，要推导过程。这样即使结论不对，你能看到模型在哪个环节出了偏差。

> 我要在 A100 80GB 上训练一个 70B 的模型，用 ZeRO-3，帮我推演显存够不够，并把计算过程展开。

### 2.3 验证与突破技巧

**技巧六：反直觉提问**

迫使模型不是复述训练数据中的常见表述，而是去碰撞矛盾点。

> 哪个关于 GPU 显存的常见理解其实是错的？

**技巧七：对立面/冲突提问**

用对立观点推动模型站边，逼出更完整的论证结构。

> 假设有人说显存带宽比显存容量更重要，你怎么反驳？反过来，什么场景下容量更重要？

**技巧八：分层追问**

不指望一次问透。换不同的维度轮番问，覆盖越立体越好。

> - 显存物理层面是怎么组织的？
> - 显存操作系统层面怎么管理？
> - 显存框架层面（PyTorch/Paddle）怎么调度的？

## 第三部分：决策框架——什么时候用什么技巧

这些技巧不是越全越好，而是取决于你的当前认知状态和想要的下一个状态。

### 3.1 先问自己三个问题

1. 我知道什么？——决定要不要设基线（Known-Knowns 声明）
2. 我想要多深的回答？——决定深度层级
3. 我拿这个回答来干嘛？——决定提问结构

### 3.2 决策矩阵

| 你的状态 | 核心目标 | 主用技巧 | 避开的技巧 |
|----------|----------|----------|-----------|
| 零基础入门 | 建立框架 | 术语锚定 + 丰富描述 | 反直觉、对立面 |
| 有基础但卡点 | 精准突破 | 声明基线 + 推导链 | 角色视角（不是必须） |
| 验证认知完整性 | 查漏补缺 | 对立面 + 反直觉 | 丰富描述（够了就不用） |
| 需要做决策 | 辅助选择 | 约束饱和 + 推导链展开 | 反直觉 |
| 需要创意/批判 | 打开思路 | 反直觉 + 对立面 + 跨域混杂 | 密集同域术语 |

### 3.3 技巧不是越多越好

过量的技巧组合会产生反效果。具体来说：

**术语的密度效应：**

- 同域术语 2-4 个：注意力收敛强化，最佳状态
- 同域 5-8 个：密集聚焦，精确但可能遗漏边界条件
- 跨域混合 3 个以上：attention 分布撕扯，有幻觉风险
- 超过 8-10 个阈值：注意力均匀稀释，输出变得含糊泛化

**实用建议：** 同域术语可以大胆堆，越多越准。跨域术语控制在 2-3 个之间，超过这个线不如拆成两次问。

## 第四部分：这些技巧跨模型通用吗？

通用。原因不在于模型训练得有多像，而在于架构层面它们共享同一个底层机制。

所有主流 LLM（GPT / Claude / Gemini / DeepSeek / Qwen 等）都基于 Transformer Decoder-only 架构。这个架构下，回答质量由两个因素决定：输入 token 的质量（对应术语精度）和 attention 能利用的上下文总量（对应约束密度）。词汇锚定和约束饱和利用的不是某个模型的独特能力，而是 Transformer 处理输入时的固有偏好。

但微妙差异仍然存在：

- Claude 对 Role Prompting 和结构化约束响应最稳定
- GPT 风格偏解释性
- Gemini 偏简洁，约束太多时可能跳过细节
- 要求展开推导链在推理模型上效果尤其显著
- 反直觉提问时 Claude 最有立场，GPT 偏中立，Gemini 偏保守

这些是训练时的 RLHF 偏好差异，不是架构差异。

## 第五部分：实操案例——KV Cache 提问对比

用一个具体案例来验证上述框架。主题选在 Transformer 推理时的 KV Cache，对同一个主题用三种完全不同的提问策略，观察输出的差异。

### 5.1 版本A：入门导向

使用的技巧：术语锚定 + 角色视角（零基础读者） + 约束饱和（类比 + 推导链 + 公式）

> 请用尽可能简单的语言解释 Transformer 自回归推理时 KV Cache 是什么、为什么需要它、以及它占用了什么资源。假设读者对 Attention 机制只有基本概念，不要假设读者知道 decoder-only 架构的细节。必须展开推导过程，用比喻或类比帮助理解，并明确指出显存占用公式。

回答的开场是一个对话场景模拟（"今天天气真好"），让读者先看到问题在哪。然后用"图书馆查资料"的比喻把 KV Cache 具象化——没有缓存的蠢做法是每写一段都把之前的书重新翻一遍，有缓存的聪明做法是书桌上的书直接看。显存公式先拆因子再套真实数字（Llama 70B 10.7GB，Llama 8B 2.1GB）。

回答特点：先让读者知道"这个技术为什么存在"，再讲"它是什么"。不假设任何前置知识。

### 5.2 版本B：精准突破导向

使用的技巧：声明已知基线 + 分层追问（4 个具体卡点） + 推导链展开

> 以下是我的基线声明：我知道 KV Cache 是在自回归解码时缓存 Key 和 Value 矩阵、避免重复计算。但我卡在几个点上：为什么只在 decoder 层做？MHA 中怎么按 head 组织内存？GQA/MQA 有什么变化？Prefill 和 Decode 阶段有什么不同？

回答跳过动机解释，直接上三张对比表（Encoder vs Decoder vs Cross-attention），然后给出 PyTorch 代码级别的内存布局说明——tensor shape、row-major 排列、每个 step 怎么 append。接着对比 MHA vs GQA vs MQA 的 Cache 结构差异，最后分析 Prefill 和 Decode 阶段的计算属性差异（compute-bound vs memory-bandwidth-bound）。

回答特点：技术密度最高。直接钻声明过的卡点，跳过所有已知的基础内容。

### 5.3 版本C：反直觉/批判验证导向

使用的技巧：反直觉提问 + 对立面 + 追边界条件

> 关于 KV Cache，最常见的误解是什么？什么场景下 KV Cache 反而不如不用？"KV Cache 是推理加速的关键"——这句话成立的前提条件是什么？长上下文下有什么工程挑战？

回答没有标准的知识点讲解，直接列了三个常见误解："KV Cache 的显存占用跟模型参数量成正比"是错的；"KV Cache 只加速不拖慢"是错的；"多 batch 只是线性增长"是低估严重性的。然后分析什么场景下 KV Cache 反而不如不用（短输出、极短序列、滑动窗口频繁重算），再列出 KV Cache 成立的五个前提条件。

回答特点：不是让模型解释知识，而是让模型输出隐藏的反面信息。适合验证自己的理解是否完整，或寻找盲区。

### 5.4 三版横评

| 维度 | 版本A（入门） | 版本B（突破） | 版本C（批判） |
|------|-------------|-------------|-------------|
| 核心技术 | 场景推导 -> 比喻 -> 公式 | 对比表 -> 代码 -> 内存布局 | 列表反转 -> 条件拆解 -> 边界 |
| 前置知识要求 | 低 | 高 | 中 |
| 输出消耗 | 约 2K tokens | 约 8K tokens | 约 3K tokens（含主动搜索） |
| 回答覆盖的知识面 | 动机 + 直觉 | 机制 + 实现 | 盲区 + 边界 |
| 最适合的场景 | 第一次接触 | 有基础但卡细节 | 以为懂了想验证完整性 |

三版的知识覆盖面几乎没有重叠。这就是"不同的提问目标 -> 选择不同的技巧组合 -> 得到完全不同的回答维度"的最好实证。

## 第六部分：一个延伸问题——术语密度与语义稀缺

在讨论过程中，还触及了一个延伸问题：如果一条 prompt 里塞了大量术语，模型会怎么处理？

这引向了向量检索的核心矛盾。在纯向量搜索（如 RAG 中的语义检索）场景下，不常见的语义在向量空间中的样本密度低，近邻搜索时最近邻也可能离得远，相似度分数低，top-N 里就会混入噪声。这不是简单的"准确性下降"，而是**召回偏斜**——倾向召回常见语义的结果，漏掉真正相关但稀疏的结果。

而且不常见语义往往涉及训练语料中的低频 token 组合，**嵌入模型本身就没见过足够多的类似表达**，表征就不准确，问题在 embedding 层就已经开始了，不是搜索阶段才暴露的。

常见的工程缓解手段包括 Hybrid Search（向量 + BM25 加权）、Multi-Query / Query Expansion（扩展语义覆盖）、以及 Reranking（cross-encoder 二次排序）。

## 第七部分：写在最后

所有技巧的底层逻辑只有一句话：**提问不是为了给模型更多信息，而是为了减少模型对你需求的猜测空间。**

模型本身是一个基于概率的生成机器。它天然倾向于输出最"常见"的路径。你的任务就是通过精准的词汇和丰富的约束，把那条"常见路径"扭成你真正想要的那条"精准路径"。

这个过程不需要你理解 Transformer 的内部数学，但如果你能用"词汇锚定 + attention 分布重心 + 约束面交集 + 自回归方差收敛"这个框架来理解它，你会发现**每一次有效的对话，本质上都是在高维向量空间中做精确制导**。
